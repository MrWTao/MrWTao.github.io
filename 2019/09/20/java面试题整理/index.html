java面试题整理

发表于 2019-09-20 | 分类于 面试

第一部分 java内存模型

1.描述一下java内存模型?

java虚拟机规范中定义了一种java内存模型JMM用来屏蔽掉各种硬件和操作系统的内存访问差异，以实现让java程序在各种平台下都能达到一致的内存访问效果。

java内存模型的主要目标是定义程序各个变量的访问规则，即在虚拟机中将变量存储到内存和从内存中取出变量这样的底层细节。

(1).java内存模型规定所有的变量都存储在主内存中，并且每条线程都有自己的工作内存，线程的工作内存中保存了被该线程使用到的变量的主内存副本拷贝，线程对所有变量的所有读写操作都必须在工作内存中进行，而不能直接操作主内存中的变量，线程之间的工作内存是隔离的，无法直接访问，需要通过主内存来传递完成；

(2).主内存与工作内存之间具体的交互协议也被java内存模型所定义。java内存模型规定以下8中操作都是原子的、不可再分的：【lock/unlock/read/load/use/assign/store/write】

  • lock（锁定）：作用于主内存的变量，标识一个变量为线程独占的状态
  • unlock（解锁）：作用于主内存的变量，释放一个处于锁定状态的变量
  • read（读取）：作用于主内存中的变量，将主内存中的一个变量传输到线程的工作内存中去
  • load（载入）：作用于工作内存中的变量，把read操作得到的变量值放入工作内存中的变量副本中
  • use（使用）：作用于工作内存中的变量，将工作内存中的变量副本的值传递给执行引擎
  • assign（赋值）：作用于工作内存中的变量，将从执行引擎中接受到的值赋值给工作内存中的变量副本
  • store（存储）：作用于工作内存中的变量，把工作内存中的变量副本的值传递到主内存中以便write操作
  • write（写入）：作用于主内存中的变量，把store操作从线程工作内存中得到的变量值放入主内存的变量中

(3).volatile关键字修饰变量的特殊规则：首先volatile关键字修饰的变量具备两个特点，可见性和有序性。可见性可以保证对volatile变量的修改会被其他线程是立即可见的，其实现方式是线程在使用volatile类型变量的时候会强制让线程工作内存中的变量父辈失效，重新去主存中获取最新的变量的值，我们可以认为volatile修饰的变量在多线程的环境下不存在一致性的问题；volatile变量的第二个语义是禁止指令重排序，指令重排序是CPU对程序代码进行的优化操作，其前提条件是会保证程序执行的最终结果和顺序执行的结果一致。但是这里的重排序所说的数据依赖性只是在单线程环境下，多线程环境下重排序就不能保证数据依赖性了，所以需要使用volatile关键字来实现正确的数据依赖传递。volatile关键字会使用内存屏障来保证数据依赖性。

内存屏障主要分为两种：Load Barrier和Store Barrier即读屏障和写屏障。

内存屏障会值屏障两侧的指令重排序；

强制把写缓冲区或高速缓冲区中的脏数据写回主内存，让缓存中相应的数据失效。

volatile的内存屏障策略是非常保守且悲观的：

在每一个volatile写之前插入StoreStore屏障，在写操作之后插入StoreLoad屏障；

在每一个volatile读操作之前插入LoadLoad屏障，在读操作之后插入LoadStore屏障。

volatile的性能：

volatile的读性能消耗与普通变量几乎没有任何区别，但是写操作由于本身需要在代码中插入内存屏障指令来保证处理器不会发生乱序执行，所以会稍微慢一些。

(4).java内存模型天然满足的一些规则，即先行发生规则Happens-Before：

  • 程序次序规则（在单线程情况下程序的执行顺序会依次进行）
  • 管程锁定规则（对一个对象的lock操作一定先于unlock操作）
  • volatile变量规则（volatile变量的写操作一定先于后续的读操作）
  • 线程启动规则（线程的start()方法的执行一定先于线程的每一个动作）
  • 线程终止规则（线程的所有操作都先于线程的终止检测）
  • 线程中断规则（对线程的interrupt()方法一定先于被中断线程捕获到中断信号）
  • 对象终结规则（对象的初始化完成一定先于它的finalize()方法）
  • 传递性（如果A先行发生于B，B先行发生于C，那么A先行发生于C）

2.java中如何保证线程安全？

线程安全的定义：当多线程访问一个对象时，如果不用考虑这些线程在运行时环境下的调度和交替执行，也不需要进行额外的同步策略，或者在调用方法进行任何其他的协调操作，调用这个对象的行为都可以获得正确的结果，那这个对象是线程安全的。

(1).互斥同步

同步是指在多线程并发访问共享资料时，保证共享数据在同一时刻只被一个或者是一些线程使用。而互斥是实现同步的一种手段。java中互斥同步的实现就是采用锁来实现，包括但不限于synchronized和JUC包下的类库。

(2).非阻塞同步

同步互斥最主要的问题就是线程阻塞和唤醒所带来的的性能问题，因此这种同步也成为了阻塞同步，这是一种悲观的并发策略，总是认为只要不去做正确的同步策略就会出现问题。而现在随着硬件指令集的发展，我们诞生了一种乐观并发策略，通俗的讲就是先进行操作，如果没有其他线程来争夺资源就操作成功，否则就进行其他的补偿策略，比如说反复重试直到成功为止。由于这种乐观的实现不需要挂起线程，所以称为非阻塞同步。这其中有一些常见的指令：

  • 测试并设置（Test-and-Set）
  • 获取并增加（Fetch-and-Increment）
  • 交换（Swap）
  • 比较并交换（Compare-and-Swap）
  • 加载链接/条件存储（Load-Linked/Store-Conditional）

CAS指令需要3个操作数，分别是内存位置的值V和旧的预期值A和新值B，当且仅当V符合旧的预期值A时，处理器用新的值B更新V的值。整个过程是一个原子操作。

(3).java中的锁优化

  • 自旋锁与自适应自旋锁
  • 锁消除
  • 锁粗化
  • 偏向锁、轻量级锁、重量级锁的膨胀过程

第二部分 Redis数据库

1.redis为什么快？

redis是一个开源的基于内存的数据结构存储系统，他可以用作：数据库、缓存和消息中间件。支持多种数据结构，包括string、hash、list、set和zset五种。

redis提供了复制、LUA脚本、LRU驱动事件、事务和不同级别的磁盘持久化策略，并且通过哨兵和集群达成了高可用的状态。

redis速度快的原因如下：

  • 完全基于内存，绝大部分请求时纯粹的内存操作，速度快，类似于HashMap一样，查找和操作的时间复杂度都是O(1)的；
  • 数据结构简单，对数据的操作也简单；
  • 采用单线程模式，避免了不必要的上下文切换和竞争条件，也不存在多线程经常导致的切换消耗CPU，不会因为加锁、解锁、死锁等消耗性能；
  • 使用多路复用的IO模式，非阻塞处理；

2.redis如何进行持久化操作？

(1).RDB

RDB持久化方案是指在指定的时间间隔内将内存中的数据集快照写入磁盘。实际的操作流程是fork一个紫禁城，先将数据集写入临时文件，写入成功之后再替换之前的文件，用二进制压缩存储。RDB是默认开启的持久化方案，生成的快照文件叫做dump.rdb。

(2).AOF

AOF持久化是以日志的形式记录服务器所处理的每一个 写、删除操作，查询操作不会记录，以文本的方式存在，文件中可以看到详细的操作记录。

RDB持久化方式比较简单，并且由于是子进程进行持久化，不会影响redis服务的性能，而AOF方式相对来说会影响一些性能；如果数据集量非常大，那么RDB方式的redis重启会快于AOF方式；但是RDB方式的数据安全性较低，会丢失某些时间段的数据，AOF就不会；AOF方式的持久化文件大，但是也提供了一定的命令合并重写功能。

3.redis如何实现高可用？

在redis中，实现高可用的技术主要包括持久化、复制、哨兵和集群，其功能分别如下：

  1. 持久化：作用就是数据备份，保证数据不会因为进程的退出而丢失；
  2. 复制：复制是高可用的基础，哨兵和集群都是在复制的基础上实现高可用的。复制主要实现了数据的多机备份以及对读操作的负载均衡和简单的故障恢复；
  3. 哨兵：在复制的基础上，哨兵实现了自动化的故障恢复；
  4. 集群：通过集群解决了写无法负载均衡以及存储能力受到单机限制的问题，实现了较为完善的高可用方案。

4.redis中数据的缓存淘汰策略？

redis.conf配置文件中会有一个配置项maxmemory <bytes>用来配置内存提供给redis的使用空间限制，0的时候表示不做限制。redis启用了最大内存空间限制的时候，如果客户端申请空间的时候超过了maxmemory的大小，就会使用缓存淘汰策略进行淘汰。redis为我们提供了一下集中淘汰策略：

  1. noeviction：当主内存的使用达到阈值的时候所有引起申请内存的命令会报错；
  2. allkeys-lru：在主键空间中，优先移除最近未使用的key；
  3. volatile-lru：在设置了过期时间的键空间中优先移除最近未使用的key；
  4. allkeys-random：在主键空间中随机移除某个key；
  5. volatile-random：在设置了过期时间的键空间中随机移除某个key；
  6. volatile-ttl：在设置了过期时间的键空间中具有更早过期时间的key移除。

5.介绍一下redis的事务机制？

事务就是可以一次执行多条命令。

redis的事务具有三个特点：

  1. 单独的隔离操作：事务中所有命令都会按顺序执行，执行过程中不会被其他客户端发来的命令打断
  2. 没有隔离级别的概念：队列中的命令没有提交之前都不会实际的被执行，因为事务提交前任何指令都不会被执行所以也就不存在事务内的查询要看到其他事务里的更新
  3. 不保证原子性：redis同一个事务中的命令如果有一条执行失败并不会回滚，其他的命令都会执行下去

6.redis集群的工作原理和注意事项？

redis集群的工作原理如下：

  1. 客户端直接与Redis节点相连，不需要中间代理层，通过对key的Hash算法映射到某个master节点上进行操作
  2. 每个master节点带有一定数量的slave节点

优势如下：

  1. 无需哨兵，如果master挂了，集群内部自动将slave升级为master节点
  2. 可以进行水平扩容
  3. 支持自动化迁移，当slave节点挂了并且对应的master节点没有slave节点时，集群会把其余master多余的slave节点迁移到没有slave的master节点上

一些问题：

批量操作的不支持，如果硬要批量操作可以使用Hashtag的方式，保证这些可以都映射到同一台机器上，如：

对于key为{foo}.student1、{foo}.student2，{foo}student3，这类key一定是在同一个redis节点上。因为key中“{}”之间的字符串就是当前key的hash tags， 只有key中{ }中的部分才被用来做hash，因此计算出来的redis节点一定是同一个!
